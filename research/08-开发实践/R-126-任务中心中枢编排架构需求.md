# R-126 任务中心中枢编排架构需求

> **文档类型**：架构需求文档 | **日期**：2026-07-01 | **状态**：待确认
> **前置文档**：R-125（dispatch-spawn 方案评估，已归档为技术子方案）

## 一、核心诉求

任务中心作为所有任务的中枢编排器，主 agent 不再直接 spawn 子 agent，而是通过任务中心提交、审核、通知。

### 架构链路

```
用户 ← 微信 → 主agent（对话+审核层）
                 ↓ 提交任务
             任务中心（编排中枢）
                 ↓ 分配 + spawn
             agent团队（执行层）
                 ↓ 完成
             任务中心（结果回写，状态=待审核）
                 ↓ 通知主agent
             主agent（质量审核）
                 → 通过 → 写概要到任务中心 + 微信通知用户
                 → 不通过 → 通过任务中心重试
```

### 与现有架构的本质区别

| 维度 | 现有架构 | 目标架构 |
|------|----------|----------|
| 任务中心角色 | 被动记录 + flag 文件中转 | **主动编排中枢** |
| 主 agent 职责 | spawn + yield + 检查 + 汇报 | **提交任务 + 审核结果 + 通知用户** |
| agent 团队触发 | 主 agent 直接 spawn | **任务中心分配 + spawn** |
| 完成回流 | completion event → 主 agent | **agent → 任务中心 → 通知主 agent 审核** |
| 质量把关 | 纪律约束（可被绕过） | **流程必经环节（不审核不算完成）** |
| 重试 | dispatch.js 自动重试（仅超时） | **主 agent 决策，通过任务中心执行** |
| 用户通知 | 主 agent yield 后汇报 | **主 agent 审核通过后通知** |

### 任务状态机（目标）

```
pending → dispatched → running → pending_review → done
                ↑           ↓          ↓
                └── retry ←──┘    rejected
                                  ↓
                              retry（主agent发起）
```

新增 `pending_review`（待审核）状态：
- agent 团队完成后，任务进入 `pending_review`，不直接 `done`
- 主 agent 审核通过后才 `done`
- 审核不通过，主 agent 标记 `rejected` 并发起 `retry`

## 二、核心角色职责

### 主 Agent（对话+审核层）
- 接收用户需求，创建任务提交到任务中心
- 接收任务中心完成通知，审核交付物质量
- 审核通过：写概要 + 微信通知用户
- 审核不通过：通过任务中心发起重试
- **不再直接 sessions_spawn**

### 任务中心（编排中枢）
- 接收主 agent 提交的任务
- 自动分配给对应 agent 团队（研究/开发）
- 调用 gateway API spawn 子 agent
- 监控任务状态（超时检测、自动恢复）
- 子 agent 完成后回写结果，通知主 agent 审核
- 记录全链路状态流转

### Agent 团队（执行层）
- 研究团队（research-lead）：调研类任务
- 开发团队（ACP Claude Code）：开发类任务
- 完成后将结果写入指定路径 + 回报任务中心
- **不直接通知用户**

## 三、设计决策（推荐方案）

### 决策①：谁负责 spawn agent 团队？

| 选项 | 优势 | 劣势 |
|------|------|------|
| A. dispatch.js 改造版 | 改动小 | 仍是外部脚本+flag 机制，不够彻底 |
| **B. server.js 自身集成** ✅推荐 | 中枢自带执行能力，状态一致性好 | server.js 改动量中等 |
| C. 独立调度进程 | 隔离性好 | 多一个进程要管理 |

**推荐 B**：任务中心 server.js 已有 HTTP 服务和 SQLite 数据库，在 server.js 中新增 `/internal/dispatch` 接口，直接调 gateway HTTP API（`POST localhost:12145/tools/invoke`）spawn 子 agent。dispatch.js 退化为超时检测+自动重试的辅助脚本。

### 决策②：主 agent 提交任务后做什么？

| 选项 | 优势 | 劣势 |
|------|------|------|
| A. yield 等待 | 即时响应 | 阻塞对话，用户无法连续操作 |
| **B. 继续对话** ✅推荐 | 不阻塞，可连续提交 | 通知有延迟（心跳周期） |

**推荐 B**：主 agent 提交任务后告诉用户"已提交到任务中心"，继续可以对话。任务完成后通过心跳检查通知主 agent。

### 决策③：任务中心如何通知主 agent 任务完成？

| 选项 | 优势 | 劣势 |
|------|------|------|
| **A. 心跳检查 completions** ✅推荐 | 最可靠，不遗漏 | 最多30min延迟 |
| B. OpenClaw completion event | 即时 | 依赖 OpenClaw event 系统，可能丢失 |
| C. server.js 主动推送 | 即时 | 需要额外机制唤醒主 agent |

**推荐 A+B 双通道**：心跳检查为主（不遗漏），completion event 为辅（即时性）。心跳检查 .task-completions.jsonl 的新记录，发现后通知主 agent 审核。

## 四、任务流转全流程（目标）

```
1. 用户发消息 → 主 agent 接收
2. 主 agent 判断需要调研/开发 → 通过 HTTP API 提交任务到任务中心
   POST localhost:8052/api/tasks { title, type, assigned_agent, expected_output }
3. 任务中心创建任务记录（status=pending）
4. 任务中心调度器检测到 pending → 调 gateway API spawn 子 agent
   POST localhost:12145/tools/invoke { tool: "sessions_spawn", args: { agentId, mode:"run", ... } }
5. 任务状态 → dispatched → running
6. 子 agent 执行 → 完成 → 写 completions.jsonl + 交付物
7. 任务中心检测到完成 → 状态 → pending_review
8. 主 agent 心跳检测到 pending_review → 读交付物 → 质量审核
9a. 审核通过 → 写概要到任务中心 → 微信通知用户 → 状态 done
9b. 审核不通过 → 标记 rejected → 任务中心重新分配 → 回到 step 4
```

## 五、改动范围预估

| 组件 | 改动内容 | 工作量 |
|------|----------|--------|
| server.js | 新增 `/internal/dispatch` 接口 + gateway API 调用 + 状态机扩展 | 中等 |
| tasks.db | 新增 `pending_review` 状态 + `review_summary` 字段 | 小 |
| dispatch.js | 简化：仅保留超时检测 + 自动重试，删除 flag 写入 | 小 |
| AGENTS.md | 主 agent 心跳流程重写：删除 flag 检查，新增 pending_review 审核 | 小 |
| 前端 | 任务看板新增"待审核"状态卡片 + 审核操作按钮 | 中等 |
