## 一、核心诉求

任务中心作为所有任务的中枢编排器，主 agent 不再直接 spawn 子 agent，而是通过任务中心提交、审核、通知。

### 架构链路

```
用户 ← 微信 → 主agent（对话+审核层）
                 ↓ 提交任务
             任务中心（编排中枢）
                 ↓ 分配 + spawn
             agent团队（执行层）
                 ↓ 完成
             任务中心（结果回写，状态=待审核）
                 ↓ 通知主agent
             主agent（质量审核）
                 → 通过 → 写概要到任务中心 + 微信通知用户
                 → 不通过 → 通过任务中心重试
```

### 与现有架构的本质区别

| 维度 | 现有架构 | 目标架构 |
|------|----------|----------|
| 任务中心角色 | 被动记录 + flag 文件中转 | **主动编排中枢** |
| 主 agent 职责 | spawn + yield + 检查 + 汇报 | **提交任务 + 审核结果 + 通知用户** |
| agent 团队触发 | 主 agent 直接 spawn | **任务中心分配 + spawn** |
| 完成回流 | completion event → 主 agent | **agent → 任务中心 → 通知主 agent 审核** |
| 质量把关 | 纪律约束（可被绕过） | **流程必经环节（不审核不算完成）** |
| 重试 | dispatch.js 自动重试（仅超时） | **主 agent 决策 + dispatch 兜底** |
| 用户通知 | 主 agent yield 后汇报 | **主 agent 审核通过后通知** |

### 任务状态机（目标）

```
pending → dispatched → running → pending_review → done
                ↑           ↓          ↓
                └── retry ←──┘    rejected
                                  ↓
                              retry（主agent发起）
```

新增 `pending_review`（待审核）状态：
- agent 团队完成后，任务进入 `pending_review`，不直接 `done`
- 主 agent 审核通过后才 `done`
- 审核不通过，主 agent 标记 `rejected` 并发起 `retry`

### 保留现有功能
- 任务中心 Web 界面的手动提交待办任务入口保留
- 用户可直接在 Web 界面创建任务

## 二、核心角色职责

### 主 Agent（对话+审核层）
- 接收用户需求，创建任务提交到任务中心（HTTP API）
- 接收任务中心完成通知，审核交付物质量
- 审核通过：写概要 + 微信通知用户
- 审核不通过：通过任务中心发起重试
- **不再直接 sessions_spawn**

### 任务中心（编排中枢）
- 接收主 agent / Web 界面提交的任务
- 自动分配给对应 agent 团队（研究/开发）
- 调用 gateway HTTP API spawn 子 agent
- 监控任务状态（超时检测、自动恢复）
- 子 agent 完成后回写结果，通知主 agent 审核
- 记录全链路状态流转

### Agent 团队（执行层）
- 研究团队（research-lead）：调研类任务
- 开发团队（ACP Claude Code）：开发类任务
- 完成后将结果写入指定路径 + 回报任务中心
- **不直接通知用户**

## 三、已确认的设计决策

### 决策①：dispatch.js 与 server.js 定位

**两者分工明确，不是替代关系：**

| 组件 | 定位 | 职责 |
|------|------|------|
| **server.js** | 编排中枢 | 接收任务提交、**执行 spawn**（调 gateway API）、状态流转、Web 界面 |
| **dispatch.js** | 监控辅助 | 超时检测、自动重试、completions 回写、**不再写 flag 文件** |

**server.js 新增能力：**
- 接收 pending 任务 → 调 `POST localhost:12145/tools/invoke` spawn 子 agent
- 检测 completions.jsonl → 标记 `pending_review`
- 提供 `/internal/dispatch` 和 `/internal/review` API 给主 agent 调用

**dispatch.js 简化：**
- 删除 `dispatchPending()`（不再写 flag）
- 保留 `processCompletions()`、`detectStaleTasks()`、`retryFailed()`
- 新增：检测 `pending_review` 超时（见决策④）

### 决策②：主 agent 提交任务后继续对话 ✅

主 agent 提交任务后告诉用户"已提交到任务中心"，不阻塞对话。

### 决策③：通知双通道 ✅

- **主通道**：心跳检查 completions / pending_review（不遗漏）
- **辅助通道**：OpenClaw completion event（即时性）

### 决策④：spawn 鲁棒性（基于历史教训）

**历史问题清单：**

| 事件 | 根因 | 后果 |
|------|------|------|
| task-0015/0016 | spawn 静默失败，无回写 | 任务卡 running 永远不 done |
| dispatch cron ×8 | LLM prompt 复杂致超时 | 全链路瘫痪 |
| task-0026 ×3 | API 限流（共用 key） | research-lead 反复失败 |
| task-0041 | flag 写了但心跳没拾取 | 任务积压 8 小时 |
| task-0016 虚报 | 子agent 谎报完成 | 3项功能只完成1项 |

**鲁棒性设计（必须在方案中实现）：**

```
spawn 调用
  → 成功：记录 agentSessionKey + dispatched_at → 状态 running
  → 失败：记录错误信息 → 状态 spawn_failed → 自动重试（max_retries）
  → 超时（gateway 无响应）：同失败处理

状态检测（dispatch.js 每2分钟）
  → running + 超过 grace period + 无活跃子agent → 标记 failed
  → failed + retry_count < max_retries → 自动重排 pending
  → failed + retry_count ≥ max_retries → 标记 failed_final → 通知主 agent

完成回写（dispatch.js）
  → 检测 completions.jsonl 新记录 → 状态 pending_review
  → 同时验证交付物文件是否存在（不存在不标完成）

pending_review 处理
  → 不自动通过，不设超时
  → 通过心跳机制要求主 agent 检查
  → 夜间主 agent 休眠时保持 pending_review，等恢复后处理
```

**API 限流应对：**
- spawn 失败时记录 HTTP status code
- 429（限流）：延迟 60 秒后重试（dispatch.js 下轮）
- 500/502：立即标记 failed，进入重试队列
- 连续 3 次 spawn 失败 → 标记 failed_final + 通知主 agent

## 四、任务流转全流程（目标）

```
1. 用户发消息 / Web 界面提交 → 主 agent / server.js 接收
2. 主 agent → HTTP API 提交任务到任务中心
   POST localhost:8052/api/tasks { title, type, assigned_agent, expected_output }
3. 任务中心创建记录（status=pending）
4. server.js 调度检测到 pending → 调 gateway HTTP API spawn
   POST localhost:12145/tools/invoke { tool: "sessions_spawn", args: { agentId, mode:"run", ... } }
   → 成功：记录 agentSessionKey → status=dispatched → running
   → 失败：记录错误 → 自动重试 / 通知主 agent
5. 子 agent 执行 → 完成 → 写 completions.jsonl + 交付物
6. dispatch.js 检测到完成 → 验证交付物 → status=pending_review
7. 主 agent 心跳检测到 pending_review → 读交付物 → 质量审核
8a. 审核通过 → 写概要到任务中心 → 微信通知用户 → status=done
8b. 审核不通过 → 标记 rejected → 任务中心重新分配 → 回到 step 4
8c. 超时未审核（2h）→ 自动通过 → 记录 auto_approved
```
