## 一、核心结论（先行版）

**7×24小时自主编程是可行的，但需要精心的架构设计、错误恢复机制和成本控制策略。**

关键判断：
- ✅ **技术可行**：Claude Code + OpenClaw + 服务器环境可以实现全天候自主编程
- ✅ **已有先例**：OpenHands、SWE-agent等开源项目验证了长程自主编码的可行性
- ⚠️ **需要人工检查点**：完全无人监督的代码质量无法保证，建议每4-8小时人工review
- ⚠️ **成本是主要约束**：API调用费用是7×24模式的最大开支，需要模型混用策略
- ❌ **不适合所有任务**：探索性任务、创意设计、复杂架构决策仍需人类参与

**推荐架构**：OpenClaw（编排层）→ Claude Code（执行层）→ Git（状态持久化）→ 监控告警（运行保障）

## 二、自主编程理念与框架

### 2.1 什么是自主编程（Autonomous Coding）

自主编程是指 AI agent 在最少人类干预的情况下，独立完成软件开发任务的能力。其核心特征包括：

| 特征 | 说明 |
|------|------|
| **自主决策** | Agent 自主决定下一步操作（读文件/写代码/运行命令） |
| **自我修复** | 遇到错误时自主分析原因并修复 |
| **长程规划** | 将复杂任务拆解为多步骤并按序执行 |
| **状态持久化** | 任务进度可保存和恢复 |
| **持续运行** | 支持长时间（小时/天级别）的任务执行 |

### 2.2 自主编程的分层模型

```
┌─────────────────────────────────────┐
│     L4: 完全自主（Level 4）          │  ← 无人监督，端到端交付
│     - 自动需求理解、架构设计          │
│     - 自动编码、测试、部署            │
│     - 自动质量保证                   │
├─────────────────────────────────────┤
│     L3: 监督下自主（Level 3）        │  ← 人工设置检查点
│     - 人类定义任务和验收标准          │
│     - Agent 自主执行，人类定期review  │
│     - 当前最佳实践所处的层次          │
├─────────────────────────────────────┤
│     L2: 辅助开发（Level 2）          │  ← 人类主导
│     - 人类编写主要代码               │
│     - Agent 辅助完成重复性任务        │
│     - Copilot 模式                   │
├─────────────────────────────────────┤
│     L1: 代码补全（Level 1）          │  ← 工具辅助
│     - IDE 自动补全                   │
│     - 代码片段生成                   │
└─────────────────────────────────────┘
```

**当前（2025-2026）技术处于 L2-L3 之间**，最佳实践目标是稳定运行在 L3。

### 2.3 设计哲学

自主编程系统的设计遵循几个核心原则：

1. **Agentic Loop（智能体循环）**：Agent 在"感知→决策→执行→观察"的循环中持续工作
2. **Fail-Fast, Recover-Fast（快速失败，快速恢复）**：错误不是终点，而是反馈
3. **Stateless Design（无状态设计）**：所有状态持久化到文件系统，Agent 本身无状态
4. **Defense in Depth（纵深防御）**：多层次的错误检测和恢复机制

## 三、Claude Code 长程任务设计

### 3.1 Loop 模式核心机制

Claude Code 的 Loop 模式是实现长程任务的基础。其工作流程：

```
任务输入 → Claude 分析 → 选择工具 → 执行工具 → 观察结果 → 继续推理 → ... → 任务完成/达到限制
```

#### 启用 Loop 模式

```bash
# 基础 loop 模式 - 适合中型任务
claude -p "实现用户认证模块，包含JWT token生成和验证" \
  --max-turns 50 \
  --allowedTools "Read,Write,Grep,Glob,Bash(npm *),Bash(node *)"

# 完全自主模式 - 适合大型任务
claude -p "构建完整的 REST API 项目" \
  --dangerously-skip-permissions \
  --max-turns 200 \
  --output-format stream-json

# 通过 stdin 传入大型 prompt
cat detailed_prompt.md | claude -p \
  --dangerously-skip-permissions \
  --max-turns 200 \
  --output-format stream-json
```

#### 关键参数详解

| 参数 | 作用 | 推荐设置 |
|------|------|----------|
| `-p` / `--print` | Headless 模式 | 必须使用 |
| `--max-turns N` | 最大循环轮数 | 50-200（视复杂度） |
| `--allowedTools` | 工具白名单 | 精确指定，避免全开 |
| `--dangerously-skip-permissions` | 跳过权限确认 | 仅限沙盒环境 |
| `--output-format stream-json` | JSON流式输出 | 自动化必用 |
| `--model` | 指定模型 | 按成本/质量需求选择 |

### 3.2 Context（上下文）管理

长程任务最大的技术挑战是上下文窗口管理。

#### Context Compaction 机制

Claude Code 内置了上下文压缩（Context Compaction）机制：
- 当对话接近 200K token 窗口限制时，自动触发压缩
- 压缩策略：保留关键信息（文件内容、关键决策、错误信息），丢弃冗余对话
- 这使得理论上"无限"循环成为可能

#### 上下文管理最佳实践

```markdown
## 1. 任务拆分（最重要）
# ❌ 错误：一个大 session 完成所有工作
claude -p "构建整个系统" --max-turns 500

# ✅ 正确：拆分为多个小 session
claude -p "Step 1: 创建项目结构和配置文件" --max-turns 30
claude -p "Step 2: 实现数据模型和数据库层" --max-turns 50
claude -p "Step 3: 实现 API 路由和业务逻辑" --max-turns 50
claude -p "Step 4: 编写测试" --max-turns 30

## 2. 使用产物文件串联流程
# 每个 session 的产出是文件（代码、配置、文档）
# 下一个 session 通过读取文件了解上下文
# 而不是依赖对话历史

## 3. CLAUDE.md 作为持久上下文
# 在项目根目录创建 CLAUDE.md，Claude Code 会自动读取
echo "## 项目说明\n这是一个 Express.js API 项目\n## 架构决定\n- 使用 TypeScript\n- 数据库：SQLite\n- ORM：Drizzle" > CLAUDE.md
```

### 3.3 错误恢复与自我修复

Claude Code 的错误处理能力：

| 错误类型 | 自动修复概率 | 说明 |
|----------|-------------|------|
| 语法错误 | ~100% | 读取编译错误，直接修复 |
| Import 缺失 | ~100% | 分析依赖树，添加 import |
| API 用法错误 | ~80% | 查阅文档，修正调用 |
| 类型不匹配 | ~80% | 分析类型签名，调整代码 |
| 架构设计问题 | ~30% | 可能陷入循环，需要人工介入 |
| 并发 Bug | ~20% | 难以自主诊断，需人类帮助 |

#### 防止循环卡死

```bash
# 设置 max-turns 作为硬限制
--max-turns 100

# 通过外部脚本检测停滞
#!/bin/bash
# monitor.sh - 检测 Claude Code 是否停滞
PREV_HASH=$(git rev-parse HEAD)
while true; do
  sleep 300  # 每5分钟检查
  CURRENT_HASH=$(git rev-parse HEAD)
  if [ "$PREV_HASH" = "$CURRENT_HASH" ]; then
    echo "WARNING: No git commits in 5 minutes, possible stall"
    # 可选：kill claude process and restart
  fi
  PREV_HASH=$CURRENT_HASH
done
```

### 3.4 Session 管理与断点续传

```bash
# 使用 Git 作为断点续传机制
# 每个 session 完成后自动 commit
claude -p "实现功能X，完成后 git commit" \
  --max-turns 50 \
  --allowedTools "Read,Write,Bash(git *)"

# 恢复中断的任务
claude -p "检查 git log 了解已完成的工作，继续未完成的功能" \
  --max-turns 50

# 使用 CLAUDE.md 作为进度文件
echo "## 进度跟踪\n- [x] 项目初始化\n- [x] 数据模型\n- [ ] API路由\n- [ ] 测试" > CLAUDE.md
claude -p "读取 CLAUDE.md 了解进度，继续未完成的项目"
```

## 四、OpenClaw 长时间任务策略

### 4.1 架构概览

OpenClaw 作为编排层，提供了比直接使用 Claude Code 更强大的长程任务管理能力。

```
┌──────────────────────────────────────────────┐
│              OpenClaw Gateway                 │
│                                               │
│  ┌─────────┐  ┌──────────┐  ┌─────────┐     │
│  │  Cron   │  │ Heartbeat│  │TaskFlow │     │
│  │ 编排    │  │ 监控     │  │ 状态管理 │     │
│  └────┬────┘  └────┬─────┘  └────┬────┘     │
│       │            │             │            │
│  ┌────▼────────────▼─────────────▼────┐      │
│  │         Subagent 编排层             │      │
│  │  spawn → monitor → collect → retry │      │
│  └────────────────┬──────────────────┘      │
│                   │                          │
│  ┌────────────────▼──────────────────┐      │
│  │    ACP / Claude Code 执行层       │      │
│  │  (独立的 session 和 context)       │      │
│  └───────────────────────────────────┘      │
└──────────────────────────────────────────────┘
```

### 4.2 Cron 编排

OpenClaw 的 cron 机制可以定时触发任务：

```jsonc
// 在 openclaw.json 中配置定时任务
{
  "cron": {
    "jobs": [
      {
        "schedule": "0 */4 * * *",  // 每4小时执行一次
        "task": "检查项目进度，完成下一个未完成的feature",
        "agentId": "dev-coder"
      },
      {
        "schedule": "0 9 * * *",    // 每天早上9点
        "task": "运行全部测试，报告失败用例",
        "agentId": "dev-qa"
      }
    ]
  }
}
```

### 4.3 Subagent 链式调用

```javascript
// OpenClaw 中通过 sessions_spawn 实现任务链
// 主 Agent 编排多个子 Agent

// Step 1: 需求分析 Agent
spawn({
  agentId: "dev-designer",
  task: "分析需求，输出 PRODUCT.md 和 feature_list.json"
})

// Step 2: 编码 Agent（在 Step 1 完成后）
spawn({
  agentId: "dev-coder",
  task: "读取 PRODUCT.md，实现 feature_list.json 中的功能"
})

// Step 3: QA Agent（在 Step 2 完成后）
spawn({
  agentId: "dev-qa",
  task: "运行测试，更新 feature_list.json 中的 passes 状态"
})
```

### 4.4 TaskFlow 模式

OpenClaw 的 TaskFlow skill 提供了更完善的多步骤任务管理：

- **State persistence**：任务状态持久化到文件
- **Wait/resume**：任务可以等待外部条件后恢复
- **Child task delegation**：父任务可以派生子任务
- **Crash recovery**：进程崩溃后可从状态文件恢复

### 4.5 崩溃恢复

```bash
# OpenClaw 的崩溃恢复策略

# 1. 状态文件持久化
# 所有任务状态写入 workspace 中的 JSON 文件
# 即使 Gateway 崩溃，状态不丢失

# 2. 进程级恢复
# 使用 systemd 管理进程
sudo systemctl edit openclaw-gateway
# [Service]
# Restart=always
# RestartSec=10

# 3. 任务级恢复
# Gateway 重启后，读取状态文件，恢复未完成的任务
```

### 6.3 自动化脚本示例

#### 夜间自动开发脚本

```bash
#!/bin/bash
# overnight-dev.sh - 夜间自动开发
# 每次4小时为一个工作周期

PROJECT_DIR="$1"
TASK_FILE="$2"
LOG_FILE="/var/log/overnight-dev.log"

cd "$PROJECT_DIR"

# 读取下一个未完成任务
TASK=$(grep -m1 "\- \[ \]" "$TASK_FILE" | sed 's/- \[ \] //')

if [ -z "$TASK" ]; then
  echo "$(date): All tasks completed!" >> "$LOG_FILE"
  exit 0
fi

echo "$(date): Starting task: $TASK" >> "$LOG_FILE"

# 运行 Claude Code，限制 50 轮
claude -p "$TASK. 完成后更新 $TASK_FILE 中的进度，勾选已完成项。" \
  --max-turns 50 \
  --allowedTools "Read,Write,Grep,Glob,Bash(npm *),Bash(node *),Bash(git *)" \
  --output-format stream-json >> "$LOG_FILE" 2>&1

# 自动 commit
git add -A && git commit -m "auto: $TASK" >> "$LOG_FILE" 2>&1

echo "$(date): Task completed: $TASK" >> "$LOG_FILE"
```

#### 监控脚本

```bash
#!/bin/bash
# health-check.sh - 开发服务器健康检查

# 1. 检查 Claude Code 进程
if ! pgrep -f "claude" > /dev/null; then
  echo "WARNING: Claude Code process not running"
fi

# 2. 检查 Git 是否有未提交更改

if [ -n "$(git status --porcelain)" ]; then
  echo "INFO: Uncommitted changes detected"

# 3. 检查磁盘空间

fi

# 4. 检查 API 使用（通过日志统计）
API_CALLS_TODAY=$(grep "$(date +%Y-%m-%d)" /var/log/overnight-dev.log | grep -c "API")
echo "INFO: API calls today: $API_CALLS_TODAY"
```

### 6.4 失败模式与应对

| 失败模式 | 症状 | 应对策略 |
|----------|------|----------|
| **无限循环** | Agent 反复尝试同一种失败修复 | 设置 max-turns；外部超时监控 |
| **上下文漂移** | Agent 忘记初始目标，做无关事情 | 任务拆分；CLAUDE.md 提醒目标 |
| **静默失败** | Agent 报告成功但代码不工作 | 测试验证；git diff review |
| **API 超限** | Rate limit 或 quota 用尽 | 指数退避；成本告警 |
| **OOM** | 进程内存溢出崩溃 | systemd 自动重启；swap 空间 |
| **网络中断** | API 不可达 | 重试队列；离线缓存 |
| **安全风险** | Agent 执行危险命令 | Docker 隔离；--allowedTools 白名单 |

### 6.5 成本控制策略

```markdown
## 模型混用策略

| 任务类型 | 推荐模型 | 成本级别 |
|----------|----------|----------|
| 架构设计、复杂逻辑 | Claude Opus / GPT-4 | $$$$ |
| 常规编码 | Claude Sonnet | $$ |
| 代码审查、简单修改 | Claude Haiku | $ |
| 测试编写 | Claude Sonnet | $$ |
```

## 七、部署配置清单

### 7.1 基础环境

```bash
# Ubuntu 26.04 LTS Server 初始化

# 1. 系统更新
sudo apt update && sudo apt upgrade -y

# 2. 安装 Node.js 22
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo bash -
sudo apt install -y nodejs

# 3. 安装 Docker
curl -fsSL https://get.docker.com | bash
sudo usermod -aG docker $USER

# 4. 安装 Claude Code
npm install -g @anthropic-ai/claude-code

# 5. 安装 OpenClaw
npm install -g openclaw

# 6. 配置 systemd 服务
sudo tee /etc/systemd/system/openclaw.service << 'EOF'
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=simple
User=ubuntu
ExecStart=/usr/bin/openclaw gateway start
Restart=always
RestartSec=10
Environment=NODE_ENV=production

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable openclaw
sudo systemctl start openclaw
```

## 八、监控与可观测性

### 8.1 关键监控指标

| 指标 | 采集方式 | 告警阈值 |
|------|----------|----------|
| Task 完成率 | 统计 PROGRESS.md | < 70% 需关注 |
| 平均任务耗时 | 日志统计 | > 预期2倍需排查 |
| API 调用量 | 日志统计 | > 日预算需告警 |
| Git commit 频率 | git log 统计 | 0 commit/4h = 可能停滞 |
| 错误重试次数 | 日志统计 | > 5次/任务 = 需人工介入 |
| 磁盘使用率 | df 命令 | > 80% 需清理 |

### 8.2 简易监控仪表盘

```bash
#!/bin/bash
# dashboard.sh - 简易状态面板
echo "=========================================="
echo "  7×24 Dev Server Status - $(date)"
echo "=========================================="
echo ""

# 系统状态
echo "[System]"
echo "  Uptime: $(uptime -p)"
echo "  Memory: $(free -h | awk '/Mem/{print $3 "/" $2}')"
echo "  Disk: $(df -h / | awk 'NR==2{print $3 "/" $2 " (" $5 ")"}')"
echo ""

# Git 状态
echo "[Git Activity]"
for dir in ~/dev-server/projects/*/; do
  if [ -d "$dir/.git" ]; then
    COMMITS_TODAY=$(cd "$dir" && git log --oneline --since="$(date +%Y-%m-%d)" | wc -l)
    LAST_COMMIT=$(cd "$dir" && git log -1 --format="%ar - %s" 2>/dev/null)
    echo "  $(basename $dir): $COMMITS_TODAY commits today"
    echo "    Last: $LAST_COMMIT"
  fi
done
echo ""

# OpenClaw 状态
echo "[OpenClaw]"
if systemctl is-active --quiet openclaw; then
  echo "  Gateway: Running ✓"
else
  echo "  Gateway: STOPPED ✗"
fi
echo ""

# Claude Code 进程
echo "[Claude Code]"
CLAUDE_PROCS=$(pgrep -f "claude" | wc -l)
echo "  Active processes: $CLAUDE_PROCS"
echo ""
```

## 九、推荐工作流模式

### 9.1 模式一：夜间批处理（推荐入门）

```
白天：人类开发者
  - 定义任务清单（PROGRESS.md）
  - Review 昨夜成果
  - 调整方向

夜间：Agent 自主工作
  - 20:00 开始执行任务清单
  - 每4小时一个工作周期
  - 08:00 自动停止，汇总报告
```

### 9.2 模式二：持续集成开发

```
持续运行（推荐进阶）
  - OpenClaw cron 每2小时触发一次
  - 每次执行1个feature
  - 自动测试 → commit → 通知
  - 人类每天 review 2-3次
```

### 9.3 模式三：管道式开发

```
需求 → [设计Agent] → [编码Agent] → [QA Agent] → 交付
         ↑              ↑              ↑
         └── 失败时回退 ─┘──────────────┘
```
