## 一、引擎能力分析

### 1.1 Cocos Creator 3.x 渲染能力上限

Cocos Creator 3.x（最新 3.8）已具备完整的 2D/3D 渲染能力：

| 能力维度 | 支持程度 | 说明 |
|----------|----------|------|
| **PBR 材质** | ✅ 完整支持 | 基于物理的渲染，支持金属度、粗糙度、法线贴图、AO 贴图 |
| **阴影系统** | ✅ Planar + ShadowMap | 支持平面阴影和 ShadowMap 两种类型，3.8 版本进一步完善 |
| **后处理** | ✅ 完整后处理管线 | 基于 RenderGraph 架构，支持自定义后处理 Pass（Bloom、DOF、SSAO 等） |
| **粒子系统** | ✅ 内置 3D/2D 粒子 | 3.7 版本优化粒子初始化性能，支持曲线控制、GPU 加速 |
| **物理引擎** | ✅ 内置 + PhysX | 内建轻量碰撞检测（Builtin）+ PhysX 物理模拟（3.1+） |
| **Shader** | ✅ Shader Graph + 手写 GLSL | 支持可视化 Shader 编辑器和手写 Shader，Forward/Deferred 双管线 |
| **骨骼动画** | ✅ Spine + DragonBones + 动画系统 | 内置 Animation 组件支持骨骼动画，官方支持 Spine/DragonBones 导入 |
| **光照** | ✅ 多光源 | 支持平行光、点光源、聚光灯、环境光、HDR 天空盒 |
| **模型格式** | ✅ glTF 2.0 | 原生支持 glTF 2.0 / GLB 导入 |

**核心结论**：Cocos Creator 3.x 的渲染管线架构已与 Unity 相当接近（Forward/Deferred 双管线、PBR、后处理、自定义 Shader），在 2D 领域甚至更优。差距主要在：生态/资产商店规模、高端渲染特性（如 Lumen/GI）、物理引擎精度（PhysX vs Havok）、编辑器功能完整度。

### 1.2 微信小游戏平台限制

#### 包体限制

| 限制项 | 规格 |
|--------|------|
| 主包大小 | ≤ 4MB |
| 整体大小（主包+分包） | ≤ 20MB（普通）/ ≤ 30MB（开通虚拟支付） |
| 单个普通分包 | 不限制大小 |
| 单个独立分包 | ≤ 4MB |
| CDN 远程加载 | 支持，推荐 90%+ 资源走 CDN |

#### 内存限制（高性能模式）

| 设备级别 | 内存限制 | 安全建议 |
|----------|----------|----------|
| iOS 低端（2G RAM，如 iPhone 6s/7/8） | 1GB | 控制在 800MB 以下 |
| iOS 中高端（3G+ RAM） | 1.5GB | 控制在 1.2~1.3GB |
| Android | 因设备差异大，建议 ≤ 1GB | 按最低配设备规划 |

#### 渲染架构

| 模式 | 说明 |
|------|------|
| **WebView（默认）** | 基于 WebKit 内核，WebGL 1.0/2.0，兼容性好 |
| **高性能模式** | 渲染调用转为渲染指令传给宿主进程，减少内存占用，提高渲染效率 |
| **高性能+模式** | 增加逻辑加速层+渲染加速层，耗时逻辑下沉原生实现 |

#### Skyline vs WebView

- **Skyline**（小程序专用，非小游戏引擎渲染）：单页面内存减少 35%，双页面减少 50%
- **注意**：Skyline 是小程序渲染方案（WXML/WXSS），小游戏引擎走的是 WebGL，两者是不同体系
- 小游戏的性能优化主要靠 **高性能/高性能+模式** + **GPU Instancing** + **合批优化**

#### WebGL 限制

- 仅支持 WebGL 1.0（部分设备 WebGL 2.0）
- iOS WebKit WebGL 2.0 存在 uniform 变量过多时绘制异常问题
- GPU Instancing、SRP Batcher 等特性需 WebGL 2.0 支持

#### 网络限制

- WebSocket 连接上限：5 个（1.7.0+）/ 1 个（1.7.0 以下）
- 进入后台超 5 秒：客户端禁止所有网络连接
- 首屏打开留存率约 75%（25% 玩家在等待首屏时流失）

### 1.3 引擎对比

| 维度 | Cocos Creator 3.x | Unity WebGL | LayaAir 3.x | Egret |
|------|-------------------|-------------|-------------|-------|
| **2D 能力** | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ 优秀 | ⭐⭐⭐⭐ 优秀 | ⭐⭐⭐ 良好 |

| **生态/社区** | ⭐⭐⭐⭐ 活跃 | ⭐⭐⭐⭐⭐ 最大 | ⭐⭐⭐ 一般 | ❌ 已停运 |

**关键差距总结**：Unity WebGL 在 3D 渲染质量上领先（更多内置 Shader、更好的 PBR 效果、更大的生态资产），但包体大是硬伤。Cocos Creator 在 2D 和小游戏原生适配上是最佳选择。

## 三、技术天花板分析

### 3.1 3D 渲染上限

| 指标 | 能做到的程度 | 限制因素 |
|------|-------------|----------|
| **光影** | PBR 材质 + 多光源 + 阴影（Planar/ShadowMap）+ 后处理 | WebGL 1.0 限制、内存预算 |
| **粒子特效** | 内置 3D 粒子系统，支持 GPU 加速 | 单发射器建议 ≤50 粒子，同屏 ≤500 粒子 |
| **角色建模** | 低-中面数 3D 模型（建议低面数 + 低质量纹理） | 包体限制、内存限制 |
| **场景规模** | 大地图 SLG 可行（三国冰河时代、无尽冬日） | 需要激进的 LOD + 合批优化 |
| **后处理** | Bloom、SSAO、DOF、色差等 | 需 WebGL 2.0，性能消耗大 |

**实测天花板**：
- 腾讯光子《最强魔斗士》：Cocos Creator 3D，3D 动作 RPG，画风接近手游品质
- 《三国：冰河时代》：Cocos 3.x，全 3D SLG，登顶畅销榜
- Unity WebGL 方面《无尽冬日》《向僵尸开炮》展示更高品质上限

### 3.2 2D 动画和特效上限

| 指标 | 能做到的程度 | 备注 |
|------|-------------|------|
| **Spine 骨骼动画** | ✅ 完整支持 | 优化：低配降至 15-20fps、静态时替换为静态图 |
| **DragonBones** | ✅ 完整支持 | 微信官方推荐方案之一 |
| **Live2D** | ✅ 完整支持 | 微信官方推荐方案之一 |
| **帧动画** | ✅ 完整支持 | 常见方案 |
| **2D 粒子特效** | ✅ 内置粒子系统 | 建议同屏 ≤500 粒子 |
| **GPU 动画化** | ✅ 可实现 | 10 万单位级别 Spine 同屏（需特殊方案） |

**案例**：
- 《永远的蔚蓝星球》（网易）：同屏 200+ 怪物，GPU 动画替代 Spine
- 《开心消消乐》：大量 Spine 动画，精细的角色表现

### 3.3 同屏性能极限

| 指标 | 推荐值 | 极限值 | 备注 |
|------|--------|--------|------|
| **同屏 3D 对象** | 100-500 | 1000+（需 GPU Instancing） | 万人同屏需特殊方案 |
| **同屏 2D 对象** | 500-2000 | 10000+（需合批） | Cocos 3.0 在 10000 三角面时掉帧 |
| **同屏粒子** | 100-300 | 500（不建议超过） | 单发射器 ≤50 |
| **DrawCall** | ≤100（移动端） | 200+（需合批） | 合批是关键优化 |
| **帧率目标** | 30fps（低端机）/ 60fps（高端机） | 看具体场景 | 微信小游戏不支持 JIT，CPU 密集操作受限 |

### 3.4 网络游戏能力

| 能力 | 支持程度 | 备注 |
|------|----------|------|
| **WebSocket** | ✅ 最多 5 个连接 | 足够多频道通信 |

### 3.5 包体限制与资源管理策略

```
推荐策略：
├── 主包（≤4MB）：引擎代码 + 首屏必要资源 + 启动界面
├── 分包 1：核心玩法资源（地图、角色、UI）
├── 分包 2-N：扩展内容（关卡、音效、特效）
└── CDN 远程加载：90%+ 资源（贴图、模型、音频）
    ├── 首次启动：仅下载主包 → 用户看到首屏（≤3秒目标）
    ├── 进入玩法：按需加载分包 + CDN 资源
    └── 资源缓存：wx.downloadFile → wx.getFileSystemManager
```

**资源优化要点**：
- 纹理压缩：ASTC（iOS）/ ETC2（Android），比 PNG 小 4-8 倍
- 模型优化：低面数建模，glTF Draco 压缩
- 音频：MP3/OGG，避免 WAV
- 合批：减少 DrawCall，使用静态合批 + 动态合批
- 内存：纹理 atlasing、按需加载/卸载、对象池

## 四、对 YOLObster 项目的意义

### 4.1 2D 横版魂系在微信小游戏上的品质定位

**YOLObster 是 2D 横版轻量魂系游戏，使用 Cocos Creator 3.x。基于调研结论：**

| 维度 | 可达品质 | 参考案例 |
|------|----------|----------|
| **角色动画** | Spine 骨骼动画，流畅的动作表现 | 《英雄没有闪》《跃动小子》的 2D 动作 |
| **场景画面** | 精美 2D 场景 + 视差滚动 + 动态光照 | 《遗弃之地》的中式横版塔防 |
| **特效** | 粒子特效 + 刀光 + 击飞效果 | Spine + 内置粒子系统足够 |
| **帧率** | 30-60fps 稳定（2D 项目性能充裕） | 2D 项目远低于 3D 性能瓶颈 |
| **包体** | 主包 4MB 足够放 2D 游戏（2D 美术体积小） | 《羊了个羊》主包极小 |
| **整体品质** | **A 级（优秀 2D 游戏）** | 参考 Cocos Store 的 2.5D 横版格斗项目 |

### 4.2 技术限制和规避方案

| 限制 | 影响 | 规避方案 |
|------|------|----------|
| **不支持 JIT** | CPU 密集计算慢（碰撞检测、寻路） | 使用引擎内置物理引擎（C++ 编译为 WASM） |
| **内存 1GB 限制** | 纹理、音频占用需控制 | ASTC/ETC2 压缩 + 纹理 atlasing + 按需加载 |
| **首屏 3 秒目标** | 25% 玩家在此流失 | 主包极简化 + 启动画面 + 分包异步加载 |
| **后台 5 秒断网** | 在线功能中断 | 断线重连 + 单机为主的设计 |
| **包体 4MB 主包** | 首批美术资源受限 | 首屏仅放 1-2 个关卡 + 分包渐进加载 |

### 4.3 品质目标建议

**YOLObster 合理品质目标：A 级（优秀 2D 游戏）**

- ✅ Spine 骨骼动画（角色 + Boss，流畅打击感）
- ✅ 2D 视差滚动场景（3-4 层视差，营造深度感）
- ✅ 粒子特效系统（刀光、火焰、暗影、死亡特效）
- ✅ 动态 2D 光照（火把、法术光源）
- ✅ 精美 UI（像素/手绘风格统一）
- ❌ 不建议追求 3D 渲染（2D 魂系风格更匹配，且性能更优）
- ❌ 不需要 PBR / HDR 等重 3D 特性

**核心策略**：2D 横版是 Cocos Creator 的舒适区，品质可以做到很高。把精力放在动画表现、打击感、关卡设计上，而不是和 3D SLG 比画质。
