## 一、聚宽平台概述

### 1.1 定位与核心功能
聚宽（JoinQuant，joinquant.com）是中国主流量化投研云平台之一，面向量化爱好者（宽客），提供：
- **精准回测引擎**：支持日线、分钟线、Tick级（VIP）回测
- **高速实盘交易接口**：支持股票、场内基金、期货的自动化实盘交易（通过合作券商）
- **易用的API文档**：基于Python的事件驱动框架
- **策略库与社区**：活跃的策略分享社区，由易入难的策略库
- **Jupyter研究环境**：云端Python2/Python3研究环境

### 1.2 数据覆盖
- **行情数据**：2005年至今完整的A股Level1数据，实时更新
- **财务数据**：上市公司资产负债、现金流、利润、财务指标数据，盘后更新
- **基金数据**：ETF、LOF、分级A/B基金、货币基金的行情和净值
- **衍生品数据**：金融期货数据
- **其他**：股票指数、行业板块、概念板块、宏观数据
- **因子数据**：价值、成长、动量、流动性、波动率、质量、规模等风格因子，以及数百个描述因子（基础、情绪、风险、每股等）

### 1.3 免费版 vs 付费版

| 功能 | 免费版 | VIP | SVIP |
|------|--------|-----|------|
| 回测数据频率 | 日线、分钟线 | + Tick级 | + Tick级 |
| 模拟交易 | 受限 | 全部实时运行 | 全部实时运行 |
| Jupyter环境内存 | 1GB | 更高 | 更高 |
| 数据延迟 | 可能T+1 | 实时 | 实时 |
| 技术支持 | 仅留言 | 高级客服 | 高级客服 |
| Tick级回测 | ❌ | ✅ | ✅ |

> ⚠️ VIP/SVIP具体价格未在官方公开页面明确标注，需登录查看或咨询客服。

### 1.4 支持市场
A股（主板、创业板、科创板）、场内基金（ETF/LOF）、金融期货。**不支持港股、美股。**

## 三、AI Agent 自动化可行性（核心重点）

### 3.1 聚宽的自动化能力现状

| 自动化场景 | 可行性 | 方案 |
|-----------|--------|------|
| 程序化获取数据 | ✅ 可行 | JQData（本地Python SDK），需付费或免费额度 |
| 程序化创建/修改策略 | ❌ 无官方API | 需浏览器自动化 |
| 程序化运行回测 | ❌ 无官方API | 需浏览器自动化 |
| 程序化获取回测结果 | ⚠️ 间接可行 | JQData获取数据 + 本地Backtrader回测 |
| 程序化运行模拟盘 | ❌ 无官方API | 需浏览器自动化 |
| Jupyter远程调用 | ⚠️ 受限 | 云端环境，无SSH/HTTP入口 |

### 3.2 三种自动化方案

#### 方案A：JQData + 本地回测（推荐 ⭐）
```
流程：AI Agent → JQData API获取数据 → 本地Backtrader回测 → 评估迭代
```
- **优势**：完全程序化，无需浏览器，AI Agent可直接操作
- **劣势**：需自行搭建回测框架，JQData免费额度有限
- **成本**：JQData免费版有限额，付费版约¥200-500/月

#### 方案B：Tushare + Backtrader（免费方案）
```
流程：AI Agent → Tushare API获取数据 → 本地Backtrader回测 → 评估迭代
```
- **优势**：Tushare开源免费，社区活跃，Tushare Skills已集成NLP
- **劣势**：数据质量和更新速度不如聚宽，高级数据需积分
- **成本**：基础免费，高级功能需积分（注册送积分）

#### 方案C：聚宽浏览器自动化（Playwright/Selenium）
```
流程：AI Agent → Playwright操控浏览器 → 登录聚宽 → 创建策略 → 运行回测 → 抓取结果
```
- **优势**：利用聚宽完整的回测引擎和可视化
- **劣势**：脆弱（页面改版即失效）、速度慢、需维护登录态
- **适用场景**：需要聚宽特有的Tick级回测或社区分享时

### 3.3 最优方案推荐

**推荐：方案A（JQData + 本地Backtrader）为主，方案B（Tushare）为备**

理由：
1. AI Agent的核心价值在于**策略生成和迭代**，不在于回测引擎本身
2. JQData提供高质量的A股数据（2005年至今），通过Python API直接调用
3. Backtrader是成熟的开源回测框架，完全可编程控制
4. 整个流程可以用代码串起来，无需人工干预

**前置条件**：
1. 聚宽账号（注册免费）
2. JQData Token（免费版有额度限制，建议升级付费）
3. 本地Python环境 + Backtrader库
4. AI Agent能执行Python代码

**预期自动化程度**：
```
✅ 自动生成策略代码（AI Agent直接写Python）
✅ 自动获取数据（JQData/Tushare API）
✅ 自动运行回测（Backtrader本地执行）
✅ 自动评估指标（夏普比率、最大回撤、年化收益）
✅ 自动迭代优化（根据评估结果调整参数/逻辑）
⚠️ 自动实盘交易（需对接券商QMT，复杂度高）
```

## 四、结合R-019巴菲特策略的可行性

### 4.1 NCAV/Net-Net策略在聚宽上的实现

聚宽的财务数据API**完全支持**实现NCAV策略：

```python
# 核心计算逻辑（伪代码）
query(
    valuation.code,
    valuation.market_cap,           # 总市值
    balance.total_current_assets,   # 流动资产合计
    balance.total_liability,        # 负债合计
).filter(
    balance.total_current_assets - balance.total_liability > valuation.market_cap
)
```

关键财务字段：
- `balance.total_current_assets` — 流动资产合计 ✅
- `balance.total_liability` — 负债合计 ✅
- `balance.total_assets` — 资产总计 ✅
- `valuation.market_cap` — 总市值 ✅

### 4.2 A股Deep Value机会分析

**积极因素**：
- A股破净股数量在市场低迷期可达200-300只，存在NCAV机会
- 微盘股在2024年表现活跃，万得微盘股指数突出
- 证监会减持新规：破发、破净公司禁止减持，政策关注度高

**风险因素**：
- ⚠️ 2024年10月起主板市值退市标准从3亿提至5亿，微盘股退市风险增大
- ⚠️ A股NCAV股票多为ST或濒临退市的小公司，流动性差
- ⚠️ 流动性压制风险：微盘股虽然涨幅大但波动剧烈
- ⚠️ A股缺乏美股式的清算/并购机制，"价值陷阱"更常见

### 4.3 建议
在A股实施NCAV策略需要**额外风控**：
1. 排除ST/*ST股
2. 排除市值<10亿的微盘股（退市风险）
3. 加入流动性筛选（日均成交额>500万）
4. 分散持仓（至少20只）

# R-020b: 聚宽浏览器自动化回测可行性评估

**评估日期**: 2026-03-31
**结论**: **部分可行，但不推荐作为主方案**

---

## 1. 聚宽网页结构分析

### 页面架构
- 聚宽是重度 SPA（单页应用），内容页通过 JS 动态渲染（web_fetch 返回"加载中"空壳）
- 策略编辑页面需要登录后才能访问，且 URL 结构不公开稳定（如 `/algorithm` 返回 404，实际入口可能通过 JS 路由）
- 社区帖子页面同样是 JS 渲染的 SPA

### 策略编辑器
- 聚宽策略编辑器基于 Web IDE（社区未明确披露是 CodeMirror 还是 Monaco，但这是主流在线代码编辑器的两种选择）
- **代码注入方式**：均可通过以下方式实现：
  - **DOM 操作**：定位编辑器容器，直接修改其内部文本节点或调用编辑器 API（如 `editor.setValue()`）
  - **剪贴板注入**：通过 Playwright 的 `fill()` 或键盘事件模拟粘贴，更稳定
  - **剪贴板方式更推荐**：不依赖编辑器内部 DOM 结构，改版影响小

### 回测交互
- 回测需要设置参数：起止日期、初始资金、基准指数、频率等
- 参数通过表单元素设置，DOM 操作可行
- 回测执行是异步的，需要轮询结果页面或等待特定 DOM 元素出现来判断完成

---

## 2. Playwright 自动化可行性

### 登录（⚠️ 关键障碍）
- **聚宽有滑块验证码**：社区已有帖子确认自动签到遇到滑块验证码
- 现有方案：
  - 保存 Cookie/LocalStorage 复用登录态（短期可行）
  - 滑块验证码逆向（维护成本高，可能违反 ToS）
  - 手动登录后导出 session（半自动）
- **结论**：完全无人值守登录不可行，需人工介入或定期刷新 Cookie

### 代码注入
- 通过 Playwright 的 `page.evaluate()` 调用编辑器 API 可行
- 或通过模拟键盘输入 + 粘贴方式更稳健

### 结果抓取
- 回测结果（夏普比率、最大回撤、年化收益等）显示在 DOM 表格中，可提取
- 收益曲线为前端图表（可能是 ECharts），可通过截图保存
- DOM 结构依赖前端实现，改版可能失效

### 反自动化措施
- 滑块验证码已确认存在
- 未发现 Cloudflare 等重型 WAF
- 但频繁自动化操作可能触发风控

---

## 3. 稳定性和维护成本

### 页面改版频率
- 聚宽作为成熟商业平台，UI 改版频率中等（估计每年 1-2 次大改）
- SPA 路由和 DOM 结构变化会导致脚本失效
- **估计每次改版需 2-4 小时修复**

### 社区经验
- 社区有人做过**自动签到**的 Selenium 自动化（但被滑块验证码打断）
- **未发现**任何人做过自动化回测的公开案例
- 说明这条路走的人少，缺乏现成参考

### API 替代
- 聚宽提供 **JQData SDK**（Python 数据接口），可本地获取数据
- 聚宽的回测 API **仅限平台内部使用**，无公开的回测 REST API
- 这正是浏览器自动化被考虑的原因——没有官方回测 API

## 4. 与方案 A（JQData + Backtrader）对比

| 维度 | 方案 A (JQData+Backtrader) | 方案 C (浏览器自动化) |
|------|---------------------------|----------------------|
| 数据准确性 | 使用同一 JQData 数据源 | 使用聚宽平台数据，一致 |
| 回测引擎 | Backtrader（开源、可控） | 聚宽内置引擎（更贴近实盘） |
| 维护成本 | 低（API 稳定） | 高（SPA 改版、验证码） |
| 开发成本 | 中 | 高（DOM 定位、异步等待） |
| 运行速度 | 快（本地计算） | 慢（等待网页渲染+回测排队） |
| 无人值守 | ✅ 可行 | ❌ 需人工处理验证码 |
| 独特性功能 | — | 聚宽模拟交易、实盘对接 |

### 关键判断
- **方案 C 的回测引擎更贴近聚宽实盘逻辑**（如撮合机制、滑点处理），这是方案 A 做不到的
- 但如果只是做回测验证，这个优势有限
- 方案 C 开发维护成本是方案 A 的 3-5 倍

## 5. 混合方案评估

**可行的混合路径**：
1. 日常开发调试：用方案 A（JQData + Backtrader）本地快速回测
2. 最终验证：用浏览器自动化把策略代码粘贴到聚宽，手动点运行回测，截图保存结果
3. 实盘对接：如果需要聚宽实盘，通过浏览器自动化管理策略

**半自动方案更实际**：
- AI Agent 生成策略代码 → 复制到聚宽 → 人工点回测 → AI Agent 读取结果截图分析
- 避开验证码和 DOM 不稳定问题

## 最终结论

**部分可行，推荐降级为半自动辅助方案。**

浏览器自动化回测在技术上可以实现大部分步骤（代码注入、参数设置、结果提取），但**登录验证码**和**SPA 不稳定**是两个硬伤，导致完全无人值守不可行。与方案 A 对比，方案 C 唯一独特优势是使用聚宽原生回测引擎（更贴近实盘），但代价是 3-5 倍的维护成本。

### 推荐实施路径
1. **主力方案**：方案 A（JQData + Backtrader），稳定可靠
2. **辅助验证**：半自动混合方案——AI 生成代码后人工粘贴到聚宽回测，AI 分析结果截图
3. **不推荐**：全自动浏览器回测方案，ROI 太低
4. **未来观察**：如果聚宽推出回测 API，重新评估

## 二、NCAV策略关键数据字段评估

NCAV（净流动资产价值）= 流动资产 - 总负债，策略需要：

| 字段 | Tushare Pro | AKShare | Baostock |
|------|------------|---------|----------|
| 流动资产合计 | ✅ balancesheet.total_cur_assets | ✅ stock_zcfz_em | ✅ 有 |
| 总负债 | ✅ balancesheet.total_liab | ✅ stock_zcfz_em | ✅ 有 |
| 总股本 | ✅ basic / daily_basic | ✅ 有 | ✅ 有 |
| 当前股价 | ✅ daily接口 | ✅ 历史行情 | ✅ 日线数据 |
| **退市股财务数据** | ⚠️ 可能仅近期 | ✅ **有专门接口** | ❓ 未确认 |

**结论**：NCAV策略对退市股数据依赖度高（历史上大量NCAV机会出现在ST/退市边缘股），AKShare的退市股接口是关键优势。

## 三、数据完整性评估

### 行情数据
- **日线OHLCV**：三个数据源均完整，停牌日处理方式不同（Tushare不输出停牌日，AKShare/Baostock可能输出空行）
- **复权因子**：Tushare最准确（有专门的adj_factor接口），AKShare支持前/后复权参数，Baostock也支持复权
- **新股上市首日**：Tushare和AKShare均能覆盖

### 财务数据
- Tushare的三大报表字段最全、最规范
- AKShare依赖东财源，字段覆盖度接近Tushare
- Baostock财务数据字段相对较少

### 已知问题
- AKShare：部分接口需Node.js环境（PyExecJS），2026年反爬更严，建议请求间隔≥3秒
- Tushare：低积分用户频控严格，恶意超频可导致Token永久封禁
- Baostock：项目维护状态不确定（2025年底仍有相关项目在用）

## 四、推荐方案

### 🏆 最佳免费数据组合

```
主力：AKShare（完全免费 + 退市股数据 + 另类数据丰富）
验证：Baostock（免费 + 交易所官方源 + 适合交叉验证）
升级：Tushare Pro 5000积分（≈500元，一次性）→ 全量无限制
```

**推荐理由**：
1. AKShare完全免费，退市股接口是NCAV策略刚需
2. Baostock数据来自交易所，作为交叉验证源提高可靠性
3. 如果预算允许，Tushare Pro 500元升级可大幅提升数据质量和API稳定性
4. 社区主流方案即"Tushare日线主力 + AKShare补充 + Baostock冷备份"

### 存储方案：Parquet + DuckDB

| 方案 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| CSV | 最简单 | 占空间大，读取慢 | 小规模测试 |
| SQLite | 单文件，SQL查询 | 行式存储，分析慢 | 小型项目 |
| **Parquet** | **列式压缩，读写快，适合回测** | 需要额外库 | **⭐ 量化回测首选** |
| Parquet + DuckDB | 高性能分析，无需数据库服务 | 略复杂 | **⭐ 推荐** |

**具体实现建议**：
```
data/
├── daily/          # 日线行情 (Parquet)
│   ├── 000001.parquet
│   └── ...
├── financial/      # 财务数据 (Parquet)
│   ├── balance_sheet.parquet
│   ├── income.parquet
│   └── cashflow.parquet
├── basic/          # 股票基础信息 (Parquet)
│   └── stock_basic.parquet
└── metadata.json   # 上次更新时间等元数据
```

### 自动更新策略

1. **初始化**：一次性拉取全量历史数据（AKShare + Baostock交叉验证）
2. **每日更新**：cron定时任务，交易日19:00后拉取当日数据
   ```bash
   # crontab示例
   0 19 * * 1-5 cd /path/to/project && python update_data.py
   ```
3. **增量逻辑**：记录每只股票最后更新日期，只拉取新数据追加到Parquet
4. **数据校验**：AKShare vs Baostock 交叉验证，OHLC偏差>1%时报警
5. **每周全量校验**：对比两源数据，修复不一致
