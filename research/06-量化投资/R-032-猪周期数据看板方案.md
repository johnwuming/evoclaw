## 摘要

本报告针对"猪周期数据自动看板"需求，调研了前端框架选型、数据存储与实时更新机制、安全防护和 systemd 部署方案。**核心推荐**：Dash (Plotly) + SQLite + cron + Nginx auth_basic + systemd，是性价比最高、落地最简单的方案。内存占用预计 200-400MB，数据更新采用 SSE 或前端轮询，安全防护采用 Tailscale VPN + Nginx 密码保护组合。

## 一、需求分析

### 1.1 用户核心诉求

- **持续运行**：数据看板不是一次性生成图片，而是 7×24 小时运行的 Web 服务
- **自动采集**：定时从 AKShare 等数据源抓取猪价、存栏量等数据
- **自动呈现**：数据更新后前端无需手动刷新，自动展示最新数据
- **轻量运行**：跑在 2-4GB 内存的腾讯云 VPS 上，无 Docker，无 GPU

## 二、前端框架选型

### 2.1 候选方案对比

| 维度 | Streamlit | Dash (Plotly) | Flask + Chart.js |
|------|-----------|---------------|-------------------|
| **部署难度** | ⭐ 极简单 | ⭐⭐ 中等 | ⭐⭐⭐ 较复杂 |
| **典型内存占用** | 200MB-2GB | 100-500MB | 50-150MB |
| **自动刷新** | st_autorefresh（轮询） | dcc.Interval（内置轮询） | 自定义 setInterval |
| **实时推送** | ❌ 无内置 | ❌ 无内置 | ✅ WebSocket/SSE |
| **图表交互性** | 良好 | **极佳**（Plotly.js 原生） | 中等 |
| **AKShare 集成** | ⭐⭐⭐ 零成本 | ⭐⭐⭐ 零成本 | ⭐⭐ 需封装 API |
| **代码量** | ~100 行 | ~150 行 | ~300+ 行 |
| **前端技能要求** | 无 | 无 | 需要 HTML/CSS/JS |
| **性能特征** | 交互时全量重跑 | 增量回调（更省资源） | 最轻量 |

### 2.2 推荐方案：Dash (Plotly)

**推荐理由**：

1. **回调机制省资源**：用户交互只触发变化的组件，不重跑整个脚本。在 2-4GB VPS 上性能稳定。
2. **dcc.Interval 内置自动刷新**：5 行代码实现定时更新指定图表，无需前端轮询。
3. **Plotly 图表交互性业界领先**：hover 详情、缩放、框选、图例筛选，对价格/存栏量等时序数据展示能力极强。
4. **AKShare 集成零成本**：akshare 返回 DataFrame，直接在回调函数中处理后传给 dcc.Graph。
5. **生产部署成熟**：Gunicorn + Nginx 方案久经考验。

**Streamlit 作为备选**：如果追求最快出原型（1-2 天内跑通），可用 Streamlit。注意控制页面数 ≤5 页，避免过度依赖交互触发的全量重跑。

**不推荐纯 Flask+Chart.js**：开发工作量是 Dash 的 2-3 倍，Chart.js 交互性不如 Plotly，对金融时序数据展示不够专业。

## 三、数据存储选型

### 3.1 SQLite vs DuckDB

**结论：SQLite**

- **数据量极小**：周度猪价数据，1年约52条，10年约520条。即使扩展到多省份/多品种，10年数据不超过 1MB。
- **SQLite 优势**：单文件、零配置、无需服务进程、直接被 Python 读写，完全满足需求。
- **DuckDB**：OLAP 列式存储，优化分析工作负载，但为此场景增加了不必要的复杂度。

### 3.2 表结构设计（参考）

## 四、数据采集与更新机制

### 4.1 cron + Python 采集脚本

**采集架构**：

```
[AKShare/数据源] → [Python采集脚本(cron触发)] → [SQLite数据库]
                                              ↓
                                   [钉钉/企业微信告警(webhook)]
```

**cron 配置示例**（每周一 08:00 执行）：

```cron
0 8 * * 1 /opt/pig-dashboard/venv/bin/python /opt/pig-dashboard/scripts/collect_pig_data.py >> /var/log/pig-collect.log 2>&1
```

**采集脚本设计要点**：

1. **重试机制**：网络请求失败时使用指数退避（exponential backoff），最多重试 3 次（间隔 2s/4s/8s）
2. **数据校验**：检查数据合理性（如猪价范围 5-50 元/公斤），异常时跳过写入并告警
3. **幂等性**：使用 `INSERT OR IGNORE` 或基于日期的唯一键避免重复写入
4. **日志**：写入结构化日志文件，便于排查

**告警机制**：

- 采集失败或数据异常时，调用钉钉/企业微信机器人 webhook 发送通知
- 可用 Python 库：`requests` + `json`

### 4.2 前端数据更新机制对比

| 机制 | 原理 | 复杂度 | 资源占用 | 推荐场景 |
|------|------|--------|----------|----------|
| **轮询（Polling）** | 前端每 N 分钟请求 API | 极低 | 中（频繁请求） | 简单场景，快速实现 |
| **SSE** | 服务器推送，单向 HTTP 通道 | 低 | 低（长连接） | **推荐**：数据更新后主动推送 |
| **WebSocket** | 全双工持久连接 | 高 | 低 | 高频实时（毫秒级） |

**推荐**：对于周度更新的猪周期数据，**SSE（Server-Sent Events）** 是最优选择：

- 基于 HTTP，无需特殊防火墙配置
- 浏览器原生支持，断线自动重连
- 服务器在数据写入 SQLite 后立即推送到所有在线前端
- 比轮询更实时，比 WebSocket 更简单

**SSE 实现示意**（Flask）：

```python
from flask import Response
import sqlite3

@app.route('/events')
def stream():
    def generate():
        # 简化示例：每 30 秒发送一次心跳 + 数据版本号
        last_version = get_db_version()
        while True:
            current_version = get_db_version()
            if current_version != last_version:
                yield f"data: refresh\n\n"
                last_version = current_version
            time.sleep(30)
    return Response(generate(), mimetype='text/event-stream')
```

**备选**：如果 SSE 实现复杂，前端每 5 分钟轮询一次 API 查询最新数据时间戳，最简单可靠。

## 五、安全防护方案

### 5.1 推荐安全栈

**方案 A（推荐大多数场景）：Tailscale VPN + Nginx auth_basic**

- Tailscale：WireGuard 协议，端到端加密，配置极简（5 分钟完成），无需开放公网端口
- Nginx auth_basic：htpasswd 密码保护，资源占用 <10MB，与任何 HTTP 服务配合良好
- systemd + UFW：服务级隔离和防火墙

**方案 B（无 VPN 依赖）**：Cloudflare Tunnel + Nginx auth_basic

### 5.2 方案详解

#### Tailscale VPN（强烈推荐）

- **安装**：curl -fsSL https://tailscale.com/install.sh | sh && tailscale up --accept-routes
- **资源占用**：后台进程约 15-30MB 内存，几乎零 CPU
- **优势**：无需开放任何公网端口，自动 NAT 穿透，免费版支持 100 台设备
- **注意**：依赖 Tailscale 中心服务器，中国大陆使用可能需要特殊配置

#### Nginx auth_basic（基础防护）

```nginx
# /etc/nginx/sites-available/pig-dashboard
server {
    listen 80;
    server_name your-vps-ip;

    location / {
        auth_basic "Pig Dashboard";
        auth_basic_user_file /etc/nginx/.htpasswd/pig-dashboard;
        
        proxy_pass http://127.0.0.1:8050;  # Dash 默认端口
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

```bash
# 生成密码文件
htpasswd -c /etc/nginx/.htpasswd/pig-dashboard admin
```

#### UFW 防火墙（补充层）

```bash
ufw default deny incoming
ufw allow 22/tcp          # SSH
ufw allow from 100.64.0.0/10 to any port 8050  # 仅允许 Tailscale IP 段
ufw enable
```

### 5.3 各方案对比

| 方案 | 安全性 | 复杂度 | 资源占用 | 推荐度 |
|------|--------|--------|----------|--------|
| Nginx auth_basic | 中 | 极简单 | <10MB | ⭐⭐⭐ 必选 |
| Tailscale VPN | 高 | 简单 | 15-30MB | ⭐⭐⭐ 首选 |
| Cloudflare Tunnel | 高 | 中等 | 30-50MB | ⭐⭐ 备选 |
| SSH 隧道 | 高 | 简单 | 极低 | 仅临时调试 |
| 防火墙 IP 白名单 | 高 | 简单 | 极低 | ⭐⭐⭐ 补充 |

## 六、systemd 服务部署

### 6.1 服务配置文件

```ini
# /etc/systemd/system/pig-dashboard.service
[Unit]
Description=Pig Cycle Dashboard
After=network.target

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/pig-dashboard
ExecStart=/opt/pig-dashboard/venv/bin/gunicorn -w 4 -b 127.0.0.1:8050 app:server --timeout 120
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

# 安全加固
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadOnlyPaths=/

[Install]
WantedBy=multi-user.target
```

### 6.2 关键命令

```bash
# 重载配置
sudo systemctl daemon-reload

# 启用开机自启
sudo systemctl enable pig-dashboard

# 启动服务
sudo systemctl start pig-dashboard

# 查看日志
journalctl -u pig-dashboard -f

# 查看状态
sudo systemctl status pig-dashboard
```

## 七、整体架构

### 7.1 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                     用户浏览器                               │
│  (通过 Tailscale VPN 或 公网IP:80/Nginx 访问)                │
└─────────────────┬───────────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────────────┐
│  Nginx (auth_basic 密码保护)                                 │
│  └── 反向代理到 127.0.0.1:8050                              │
└─────────────────┬───────────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────────────┐
│  Dash 应用 (Gunicorn 4 workers)                             │
│  ├── /          → 主看板页面 (dcc.Graph 图表)                │
│  ├── /api/data → JSON API (最新数据)                        │
│  └── /events   → SSE 流 (数据更新事件)                      │
└────────┬────────────────────┬───────────────────────────────┘
         │                    │
         ▼                    ▼
┌─────────────────┐  ┌─────────────────────────────────────┐
│  SQLite 数据库   │  │  cron: 每周一 08:00                  │
│  (pig_price)    │  │  Python 采集脚本                     │
│                 │  │  ├── AKShare 获取猪价数据            │
│                 │  │  ├── 写入 SQLite                     │
│                 │  │  └── 失败 → 钉钉/企微告警             │
└─────────────────┘  └─────────────────────────────────────┘
```

### 7.2 数据流

1. **写入路径**：cron 定时触发 Python 脚本 → AKShare 获取数据 → 写入 SQLite → 更新 Dash 内存状态
2. **读取路径**：用户浏览器 → Nginx 认证 → Dash 应用 → SQLite 查询 → Plotly 图表渲染
3. **更新推送**：SQLite 写入后 → Dash 触发 SSE 推送 → 前端收到事件 → 自动刷新图表

## 十、推荐方案总结

### 技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| **前端框架** | Dash (Plotly) | 回调机制省资源，dcc.Interval 内置刷新，Plotly 图表交互最强 |
| **数据库** | SQLite | 数据量极小，零配置，无需服务进程 |
| **数据采集** | cron + Python 脚本 | 技术成熟，可靠性高 |
| **实时更新** | SSE 或前端轮询 | 周度数据，SSE 最优，前端每5分钟轮询作为备选 |
| **安全防护** | Tailscale VPN + Nginx auth_basic | 高安全+低复杂度，零公网暴露 |
| **进程管理** | systemd | 开机自启，崩溃重启，统一日志 |

### 预计开发时间

| 任务 | 工作量 |
|------|--------|
| 环境搭建（依赖安装、Nginx、Tailscale） | 0.5 人天 |
| SQLite 表设计和采集脚本 | 0.5 人天 |
| Dash 主看板搭建（含 Plotly 图表迁移） | 1-2 人天 |
| SSE 实时推送集成 | 0.5 人天 |
| systemd 服务部署 + 安全加固 | 0.5 人天 |
| **合计** | **约 3-4 人天** |

### 不推荐方案

- **Grafana**：设计用于基础设施监控，需要额外数据库（MySQL/InfluxDB），架构过重
- **Metabase/Redash**：企业级 BI 工具，功能冗余，资源占用高（建议 2CPU + 2-4GB RAM）
- **纯 Flask+Chart.js**：开发工作量是 Dash 的 2-3 倍，图表交互性更弱
