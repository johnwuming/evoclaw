## 一、问题诊断

### 1.1 当前现状

截至 2026-06-28 08:34，Dashboard 中有 **3 个 dev 类型任务同时处于 running 状态**，全部指派给同一个 `claude-code-local`：

| Task ID | 标题 | 指派时间 |
|---------|------|----------|
| task-0008 | 合并agent状态 | 08:24 |
| task-0010 | 开发团队多任务并发控制 | 08:34 |
| task-0011 | 增加下拉刷新功能 | 08:34 |

### 1.2 根因分析

| 问题 | 影响 | 严重度 |
|------|------|--------|
| **无并发限制** — Dashboard dispatch 不检查 agent 当前负载 | 同一 agent 被 spam 多个任务，资源争抢，质量下降 | 🔴 高 |
| **无任务队列** — 所有 dispatched 任务立即变 running | 无法排队等待，无法按优先级调度 | 🔴 高 |
| **dev-lead 被绕过** — 任务直接 spawn 给 claude-code-local，不走 dev-lead 编排 | Flow A/B/C/D 流程形同虚设，Gate 1/2/3 质量门禁被跳过 | 🔴 高 |
| **无 capacity 信号** — 主 agent 心跳不知道 agent 容量 | 心跳继续 dispatch 新任务，雪上加霜 | 🟡 中 |
| **任务状态无自动回收** — agent 完成后不自动更新 task status=done | running 状态堆积，无法判断真实负载 | 🟡 中 |

### 1.3 质量风险

当多个任务同时压给一个 dev agent 时：
1. **上下文污染** — 一个 session 里混入多个无关功能的上下文，代码冲突概率激增
2. **Clean State 破坏** — dev-coder 被迫在一个 session 做多个功能，违反"一个 session 一个 feature"铁律
3. **QA 被跳过** — 压力下 dev-coder 倾向于"快速完成"，不等待 dev-qa 独立验证
4. **Regression 风险** — 多任务并行修改同一代码库，回归测试覆盖不足

## 二、设计方案

### 2.1 设计原则

1. **dev-lead 是唯一入口** — 所有 dev 任务必须经过 dev-lead 编排，不允许直接 spawn dev-coder/dev-qa
2. **串行为王** — 开发团队默认串行执行，同一时间只有一个功能在 dev-coder 手上
3. **队列优先级** — 多任务到来时进队列，按优先级 + 创建时间排序
4. **质量门禁不妥协** — 无论任务多急，Flow B 的 Gate 1/Gate 2 不可跳过

### 2.2 并发控制模型

#### Agent 容量定义

```
Agent Capacity Model
─────────────────────────────────────
Agent           | Max Concurrent | 说明
─────────────────────────────────────
dev-lead        |       1        | 一个编排 session
dev-coder       |       1        | 一个功能一个 session（铁律）
dev-qa          |       1        | 一个验证 session
dev-designer    |       1        | 一个初始化 session
research-lead   |       2        | 可并行两个调研
main (编排)      |       3        | 最多同时跟踪3个子任务
─────────────────────────────────────
```

> **关键决策**：dev team 全部 max=1。这不是性能优化，而是质量保障。dev-coder 一个 session 做一个 feature 是 CATASTROPHIC 级别的铁律，并发=1 从根本上保证这个约束。

#### 任务状态机扩展

```
pending → queued → dispatched → running → done
                ↘                ↘
                 cancelled        failed → retried (≤ max_retries)
                                  ↘
                                   paused
```

新增 `queued` 状态：
- **pending**: 刚创建，未被批准执行
- **queued**: 已批准执行，在队列中等待 agent 空闲（新增）
- **dispatched**: 已 spawn，agent 正在初始化
- **running**: agent 正在执行

### 2.3 调度算法

```python
def dispatch_next_task():
    """主 agent 心跳/cron 调用"""
    # 1. 检查各 agent 当前 running 数量
    for agent in AGENTS:
        running_count = count_tasks(status='running', assigned_agent=agent)
        if running_count >= CAPACITY[agent]:
            continue  # 该 agent 满载，跳过
        
        # 2. 取出最高优先级的 queued 任务
        task = query_one("""
            SELECT * FROM tasks 
            WHERE status = 'queued'
              AND assigned_agent = ?
            ORDER BY priority ASC, created_at ASC
            LIMIT 1
        """, agent)
        
        if task:
            # 3. 特殊处理：dev 类型任务走 dev-lead
            if task.type == 'dev':
                task.assigned_agent = 'dev-lead'
            
            # 4. spawn agent
            spawn_and_update(task)
```

### 2.4 dev 任务的路由规则

```
┌─────────────────────────────────────────────────┐
│  新 dev 任务到达                                   │
│                                                   │
│  1. Dashboard 创建 task (status=queued)            │
│  2. 检查 dev-lead 当前 running 数量                 │
│     ├─ < 1 → 改 status=dispatched, spawn dev-lead  │
│     └─ = 1 → 保持 queued，等待当前任务完成           │
│  3. dev-lead 内部按 Flow A/B/C/D 调度               │
│     └─ dev-lead 自己 spawn designer/coder/qa       │
│        （这些子 spawn 不占 Dashboard 级别 slot）     │
└─────────────────────────────────────────────────┘
```

> **核心洞察**：Dashboard 层面只看 dev-lead 的并发数。dev-lead 内部的 designer/coder/qa 由 dev-lead 自己串行编排，Dashboard 不需要感知。

### 2.5 与现有 Dashboard V4 的集成方案

#### 数据库变更（增量迁移）

```sql

ALTER TABLE tasks ADD COLUMN queue_position INTEGER DEFAULT 0;
ALTER TABLE tasks ADD COLUMN capacity_group TEXT DEFAULT 'default';
-- capacity_group: 'dev' | 'research' | 'ops' — 同组内串行

-- agent_status 表新增字段
ALTER TABLE agent_status ADD COLUMN max_capacity INTEGER DEFAULT 1;

-- 初始化容量
UPDATE agent_status SET max_capacity = 1 WHERE agent_id LIKE 'dev-%';
UPDATE agent_status SET max_capacity = 2 WHERE agent_id = 'research-lead';
UPDATE agent_status SET max_capacity = 3 WHERE agent_id = 'main';
```

#### API 变更

```

  → 如果 agent 空闲：返回 200 (dispatched)
```

### 2.6 主 Agent 侧的纪律增强

主 agent（AGENTS.md）需要增加以下规则：

```markdown
## 任务并发控制纪律

### Dispatch 前检查
spawn 任何子 agent 前，必须：
1. 查 Dashboard API `/api/capacity` 或读 tasks.db
2. 检查目标 agent 当前 running 任务数
3. 如果已达 max_capacity → 不 spawn，改入队列

### dev 任务的路由
- ✅ dev 任务 → spawn dev-lead（由 dev-lead 内部编排）
- ❌ dev 任务 → 不直接 spawn dev-coder / dev-qa
- 例外：主 agent 自己做的轻量 HTML/脚本修改不走 dev team

### 心跳时的队列检查
心跳 Step 2 增加：
- 读 queued 状态任务
- 检查对应 agent 是否空闲
- 空闲则自动 dispatch
```

### 2.7 质量保障清单

每个 dev 任务必须走完以下清单才能标记 done：

```
☐ Flow A 完成（如新项目）：PRODUCT.md + feature_list.json + init.sh + progress.md
☐ dev-coder 实现功能，git commit
☐ dev-qa Gate 1: Feature Gate — 所有 steps 通过
☐ dev-qa Gate 2: Regression Gate — 无回归
☐ dev-designer 更新 PRODUCT.md 功能状态
☐ progress.md 更新
☐ Clean State: git status clean
```

**强制执行机制**：
1. Dashboard 的 task status 只能由 dev-lead 报告完成（不是 coder）
2. dev-lead 报告完成时必须附带 QA 验证摘要
3. 主 agent 验证交付物后才更新 task status=done

## 五、总结

| 维度 | 现状 | 方案后 |
|------|------|--------|
| 并发控制 | 无，同一 agent 可被 spam 多任务 | 严格容量限制，dev=1, research=2 |
| 任务路由 | 直接 spawn 具体执行者 | dev 任务统一走 dev-lead 编排 |
| 质量保障 | 流程存在但可被绕过 | Dashboard level 强制 status check |
| 队列管理 | 无 | queued 状态 + 优先级排序 |
| 可观测性 | 只看到 running/done | 实时容量 + 队列长度 + agent 负载 |

**核心设计哲学**：开发团队的串行不是性能瓶颈，而是质量保障。每个功能独占 dev-coder 的完整注意力，经 dev-qa 独立验证，才是交付质量的根本保证。
