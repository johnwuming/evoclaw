### 1.3 MiniMax

|------|------|--------|-----------------|------|

## 二、Agent 角色能力需求分析

### 2.1 编排角色（Orchestrator）

| Agent | 核心需求 | 优先级 |
|-------|----------|--------|
| **research-lead** | 强指令遵循、复杂流程编排、JSON 解析、多步迭代判断、结果整合 | 🔴 指令遵循 > 工具调用 |
| **dev-lead** | 项目管理、任务拆解、进度追踪、结果回收、团队协调 | 🔴 指令遵循 > 结构化输出 |

**共同特点**：不做实际搜索/编码，但需要精确理解复杂指令并可靠执行流程。输出以文本决策为主。

### 2.2 搜索/工具角色

| Agent | 核心需求 | 优先级 |
|-------|----------|--------|
| **research-searcher** | web_search/web_fetch 工具调用、信息提取、JSON 输出 | 🔴 工具调用 > 信息提取 |
| **research-reviewer** | 逻辑推理、准确性判断、长上下文理解（审阅大量 findings） | 🔴 推理能力 > 上下文长度 |
| **research-citation** | URL 验证、精确文本处理、格式标准化 | 🟡 精确性 > 推理深度 |

### 2.3 开发角色

| Agent | 核心需求 | 优先级 |
|-------|----------|--------|
| **dev-designer** | 需求拆解、结构化输出（JSON）、产品设计思维 | 🟡 结构化输出 > 编码 |
| **dev-coder** | 强编码能力、代码理解与生成、debug | 🔴 编码能力 > 上下文 |
| **dev-qa** | 严谨逻辑、工具调用（exec/browser）、测试设计 | 🔴 工具调用 > 逻辑严谨 |

## 三、模型-角色匹配推荐

### 推荐方案（总览）

| Agent | 推荐模型 | 理由 |
|-------|----------|------|
| **research-lead** | `glm-5-turbo`（默认） | 编排角色需要 Agent 工作流优化，glm-5-turbo 专为此设计 |
| **research-searcher** | `kimi/kimi-code` | 搜索员需要频繁工具调用 + 信息提取，kimi-code 优化了工具调用链 |
| **research-reviewer** | `glm-5` | 审核需要最强推理能力，glm-5 旗舰模型推理最可靠 |
| **research-citation** | `glm-5-turbo`（默认） | 任务简单（URL 验证+格式化），默认模型足够，避免浪费 |
| **dev-lead** | `glm-5-turbo`（默认） | 编排角色同 research-lead |
| **dev-designer** | `kimi/kimi-code` | 结构化输出需求，Kimi 在 Agent 任务上表现好 |
| **dev-coder** | `kimi/kimi-code` ⭐ | 编码核心场景，kimi-code 专为编码优化，是最高优先级匹配 |
| **dev-qa** | `glm-5-turbo`（默认） | 需要工具调用 + 逻辑判断，默认模型已足够 |

### 推荐方案（详细理由）

#### ⭐ 核心推荐：dev-coder → kimi-code

编码是整个开发团队的价值核心。kimi-code 专为代码生成优化，作为 dev-coder 的模型可以最大化编码质量。这是整个方案中 ROI 最高的单项分配。

#### research-searcher → kimi-code

搜索员需要执行多轮工具调用链（搜索→筛选→再搜索→提取），Kimi K2.5 在 Agent 任务 benchmark 上达到开源 SoTA，kimi-code 应继承了这一优势。

#### research-reviewer → glm-5

审核员需要最高质量的事实判断和逻辑推理。glm-5 是旗舰模型，推理能力最强。虽然成本更高，但审核员调用量少（只在最终阶段），值得投入。

#### 其余角色 → glm-5-turbo（默认）

编排角色（lead）和简单工具角色（citation、qa）使用默认模型即可满足需求。glm-5-turbo 本身就是为 Agent 工作流优化的，编排任务正是其擅长场景。

## 四、成本与性能权衡

### 4.1 模型成本排序（估算，从低到高）

1. `glm-5-turbo` — 低成本（默认，已验证可行）
2. `kimi/kimi-code` — 中等
3. `MiniMax-M2.7` — 中等（非高速版稍贵）
4. `glm-5` — 高成本（旗舰）
5. `glm-5v-turbo` — 视觉附加成本

### 4.2 推荐配置的总体成本影响

当前所有 agent 均使用 `glm-5-turbo`。推荐方案仅将 3 个角色切换到其他模型：
- `research-reviewer` → `glm-5`：调用量少，成本增加有限
- `research-searcher` → `kimi-code`：调用量中等，成本适中增加
- `dev-coder` → `kimi-code`：调用量大，这是主要成本增量

**总体评估**：成本增幅约 10-20%，但核心场景（编码 + 审核）质量显著提升。
