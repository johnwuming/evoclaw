## 一、agentDir vs workspace 职责区别（任务1答案）

| 属性 | agentDir | workspace |
|------|----------|-----------|
| **定义** | agent 的提示词和元数据目录 | agent 的运行时工作目录 |
| **典型路径** | `/root/.openclaw/agents/<agent-id>/agent` | `/root/.openclaw/workspace-<team>` |
| **存放内容** | AGENTS.md（提示词）、系统提示、agent 专属配置 | 任务数据、中间产物、输出文件 |
| **持久性** | 长期存在，修改需手动编辑 | 任务相关，可能被清理 |
| **示例** | `research-lead` 的提示词在 `…/agents/research-lead/agent/AGENTS.md` | `research-lead` 的工作在 `…/workspace-research/` |

> **核心区别**：`agentDir` 是"定义agent是谁"的元数据目录，`workspace` 是"agent干活时用的"工作目录。提示词文件（AGENTS.md）应存在于 agentDir 中。

## 五、关键不一致问题汇总

### 问题 1：5个 agent 缺少 AGENTS.md 文件（高优先级）

**受影响的 agent**：main、research-lead、research-searcher、research-reviewer、research-citation

**现状**：这些 agent 在 `openclaw.json` 中配置了 `agentDir`，但该路径下**不存在 AGENTS.md 文件**。

**风险**：
- 如果 OpenClaw 依赖 AGENTS.md 加载 agent 提示词，这些 agent 可能使用了错误或过时的提示词
- research-lead 的 AGENTS.md 目前是通过 project context 注入的（`workspace-research/AGENTS.md`），而非从 agentDir 读取——这是一个**非标准路径**，容易在系统升级后失效

**建议**：为这5个 agent 创建 `/root/.openclaw/agents/<agent>/agent/AGENTS.md`，内容从现有的 project context 文件中提取。

### 问题 2：dev-lead agentDir 路径配置错误（中优先级）

**现状**：`openclaw.json` 中 `dev-lead` 的 `agentDir` 为 `/root/.openclaw/agents/dev-lead/agent`，但实际 AGENTS.md 在 `/root/.openclaw/agents/dev-lead/AGENTS.md`（上级目录）。

**dev-designer、dev-coder、dev-qa** 同样配置为 `/agent` 子目录，但实际文件在上级目录。

**影响**：OpenClaw 均可能在错误的路径查找提示词。

**修正方案**：将 `openclaw.json` 中这四个 agent 的 `agentDir` 改为去掉 `/agent` 后缀：
```json
"agentDir": "/root/.openclaw/agents/dev-lead",      // 原：…/dev-lead/agent
"agentDir": "/root/.openclaw/agents/dev-designer", // 原：…/dev-designer/agent（已无 /agent）
"agentDir": "/root/.openclaw/agents/dev-coder",     // 原：…/dev-coder/agent
"agentDir": "/root/.openclaw/agents/dev-qa",        // 原：…/dev-qa/agent
```

### 问题 3：/root/.openclaw/workspace/shared/results/ 目录不存在（中优先级）

**确认发现**：`/root/.openclaw/workspace/shared/results/` 在调研开始时**不存在**，write 工具在写入最终报告时**自动创建了该目录**。

**建议**：在系统初始化时显式创建此目录，或在 openclaw.json 中添加初始化逻辑。

## 七、统一规范文件可行性分析（任务3）

### 7.1 OpenClaw 是否支持共享引用机制？

根据现有配置和文档，**OpenClaw 没有内置的"多个 AGENTS.md 引用同一个路径规范文件"的机制**。每个 AGENTS.md 是独立的文本文件，路径以字面量形式硬编码。

可能的查阅方向（未在本次调研中确认）：
- `openclaw.json` 的 `agents.defaults.workspace` 是否支持变量插值
- 是否有 `include` 或 `reference` 机制用于 AGENTS.md

**结论**：当前版本（v2026.4.2）不支持自动路径引用。

### 7.2 替代方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. PATHS.md 共享文件** | 所有 AGENTS.md 引用 `../PATHS.md` | 简单，无需工具支持；一个改动全局生效 | 需要约定相对路径规则；OpenClaw 不感知，需手动同步 |
| **B. Symlinks** | 在各 agent workspace 中创建 symlink 指向统一路径 | 对 OpenClaw 透明 | 文件系统层面操作；删除/移动有风险；跨文件系统 symlink 有局限 |
| **C. openclaw.json 变量** | 在 `agents.defaults` 中定义路径模板，AGENTS.md 用变量引用 | OpenClaw 原生支持 | 目前不确定是否支持变量插值；需要功能开发 |
| **D. 模板生成脚本** | 用脚本从单一源生成各 AGENTS.md | 完全可控；可以加入验证 | 需要额外的构建/部署步骤；增加复杂度 |

### 7.3 推荐方案

**推荐方案 A（PATHS.md）+ 预留方案 C（openclaw.json 变量）**

理由：
- 方案 A 零工程成本，立即可实施
- 可以创建 `/root/.openclaw/paths/PATHS.md`，内容如：
  ```markdown
  # OpenClaw 全局路径规范

  ## Workspace 路径
  - main: /root/.openclaw/workspace
  - research: /root/.openclaw/workspace-research
  - search: /root/.openclaw/workspace-search
  - reviewer: /root/.openclaw/workspace-reviewer
  - citation: /root/.openclaw/workspace-citation
  - dev: /root/.openclaw/workspace-dev

  ## 共享路径
  - shared/results: /root/.openclaw/workspace/shared/results

  ## 研究子目录
  - research-plan: /root/.openclaw/workspace-research/research/research-plan.json
  - knowledge-base: /root/.openclaw/workspace-research/research/knowledge-base.json
  - gaps: /root/.openclaw/workspace-research/research/gaps.json
  ```
- 各 AGENTS.md 可以写"路径规范请参考 `../PATHS.md`"并引用具体路径

**次选方案 B（Symlinks）**：适用于 `/root/.openclaw/workspace/shared/results/` 这种共享目录——在需要访问共享结果的 agent workspace 中创建 symlink。
