> 基于 R-102a（Deep Research 产品与开源方案）和 R-102b（架构与质量技术）调研，结合 OpenClaw 研究团队现状，制定升级方案。

## 一、现状诊断

### 当前架构
```
main → research-lead → research-searcher (并行×N)
                      → research-reviewer
                      → research-citation
```

### 已验证的能力
- ✅ 角色分工（Planner→Searcher→Reviewer→Citation）符合行业最佳实践
- ✅ web_search + web_fetch 已配置可用（parallel-free）
- ✅ 智谱 MCP 三源（web-search-prime / zhipu-reader / zhipu-vision）
- ✅ JSON 结构化状态管理（research-plan.json / knowledge-base.json / gaps.json）
- ✅ 实时搜索能产出高质量报告（R-101 v2：12次搜索+30来源）

### 核心问题（按严重程度排序）

| # | 问题 | 严重度 | 根因 | 影响 |
|---|------|--------|------|------|
| 1 | **Context Overflow** | 🔴 严重 | research-lead 收集所有搜索员结果后上下文爆炸，无法写报告 | 任务静默失败，需拆分重跑 |

| 3 | **Reviewer 形同虚设** | 🟡 中等 | Reviewer 只有 read+write 权限，无法搜索验证，实际只做格式检查 | 报告质量依赖搜索员单次准确性 |
| 4 | **无迭代搜索能力** | 🟡 中等 | 搜索员一次性执行，Reviewer 发现缺口后无法触发补充搜索 | 知识覆盖不完整 |
| 5 | **搜索源单一** | 🟡 中等 | 仅 parallel-free 一个源，无交叉验证 | 信息偏颇风险 |
| 6 | **无引用验证** | 🟢 低 | Citation agent 无法验证URL可达性和内容匹配 | 引用可能失效 |

## 二、升级方案

### 升级 1：搜索结果截断（解决 Context Overflow）
**问题对应**: #1 #2 | **难度**: 低 | **预期收益**: 高

**当前**：搜索员 web_search 返回完整结果（含大段网页正文），3-4次搜索后 context 达 70k+ token。

**方案**：在 research-searcher 的 AGENTS.md 中增加输出约束：
- 每条搜索结果只提取：标题、URL、3-5句关键发现、关键数据点
- 每次搜索产出结构化摘要（非原文复制），≤500 token/条
- research-lead 收到的是精炼摘要而非原始搜索结果

**预期效果**：research-lead 可处理 15-20 次搜索结果而不溢出（从当前 4-5 次上限提升 4 倍）。

**参考**：OpenSearch 的三层压缩策略 — 截断工具输出 + 历史摘要压缩 + 保留最近N条原始消息。

---

### 升级 2：搜索源多元化（解决单一来源）
**问题对应**: #5 | **难度**: 中 | **预期收益**: 中

**当前**：仅 parallel-free（免费，可用但结果质量不稳定）。

**方案**：配置 3 层搜索源：
1. **primary**: parallel-free（已配置，零成本）
2. **secondary**: 智谱 web-search-prime MCP（已配置，中文质量好）
3. **tertiary**: Tavily API（AI原生设计，引用元数据+relevance score，1000次/月免费）

**实施**：
- Tavily 注册免费 key，通过 mcporter 或 MCP server 配置
- 搜索员 AGENTS.md 更新策略：关键数据点用 2 个源交叉验证，一般信息用 1 个源

**参考**：Tavily 在 SimpleQA benchmark 93.3% 准确率，且提供 relevance score 可用于结果筛选。

---

### 升级 3：Reviewer 强化（解决质量把控）
**问题对应**: #3 #4 | **难度**: 中 | **预期收益**: 高

**当前**：Reviewer 只能 read+write，无法搜索验证。

**方案**：
- 给 research-reviewer 增加 `web_search` 权限
- Reviewer 发现问题时，可直接搜索验证关键事实，而非只做格式检查
- 引入四级置信度分类（参考 SemanticCite）：Supported / Partially Supported / Unsupported / Uncertain
- 对 high-priority findings 要求至少 2 个独立来源确认

**注意**：Reviewer 的搜索次数需限制（3-5次），避免 context overflow。

---

### 升级 4：结构化状态持久化（解决上下文传递）
**问题对应**: #1 | **难度**: 中 | **预期收益**: 高

**当前**：research-lead 依赖完整对话历史传递信息给子 agent。

**方案**：引入 `research-state.json` 作为 agent 间的状态传递载体：
```json
{
  "task": "研究主题",
  "phase": "searching|reviewing|writing",
  "queries_executed": ["query1", "query2"],
  "findings": [
    {
      "id": "F001",
      "content": "关键发现摘要（≤3句）",
      "sources": ["url1", "url2"],
      "confidence": "supported|partial|unsupported|uncertain",
      "verified_by": ["searcher1", "reviewer"]
    }
  ],
  "gaps": ["未覆盖的问题"],
  "next_actions": ["补充搜索方向"]
}
```

**预期效果**：
- research-lead 不再需要完整搜索结果在上下文中，只读 state 文件
- 搜索员产出直接写入 state 文件，research-lead 只收集精简引用
- 即使 context 溢出，agent 可从 state 文件恢复

---

### 升级 5：参考 STORM 的多视角提问
**问题对应**: #4 | **难度**: 高 | **预期收益**: 中

**当前**：research-lead 的 Phase 1 规划拆解为子问题，但视角单一。

**方案**：在 Phase 1 增加多视角分析步骤：
- 分析研究主题，识别 2-3 个不同视角（如：技术视角、商业视角、用户视角）
- 每个视角独立生成搜索查询
- 搜索员按视角分配任务

**参考**：Stanford STORM 的多视角提问策略 — 模拟 Wikipedia 编辑与领域专家的对话，FreshWiki 测试覆盖广度提升 10%。

---

### 升级 6：Jina Reader 轻量抓取
**问题对应**: #2 | **难度**: 低 | **预期收益**: 中

**当前**：全文抓取依赖 browser（重）或 web_fetch（禁用）。

**方案**：搜索员发现关键 URL 后，用 Jina Reader（`r.jina.ai/URL`）快速获取正文，比 browser 轻量 10 倍，零配置。

**实施**：在搜索员 AGENTS.md 中增加工具使用指引：
```
URL 正文抓取优先级：
1. web_search 结果摘要（直接用，不需要额外请求）
2. Jina Reader: curl r.jina.ai/<URL>（轻量全文）
3. browser（最后手段，需启动 Chromium）
```

## 四、预期效果对比

| 维度 | 当前 | 升级后 |
|------|------|--------|
| 单任务搜索容量 | 4-5次（context 溢出） | 15-20次（截断+state文件） |
| 搜索源数 | 1个 | 3个（parallel-free + 智谱 + Tavily） |
| 引用验证 | 无 | 四级置信度 + URL 可达性检查 |
| Reviewer 实质 | 格式检查 | 事实核查 + 补充搜索 |
| 迭代搜索 | 不支持 | state 文件驱动，可恢复/补充 |
| 报告质量 | 依赖单次准确性 | 多源交叉 + Reviewer 独立验证 |
